# Internationalization (i18n)

The desktop frontend is fully i18n-ized: every user-facing string lives in a message catalog and resolves at runtime, so
the app is translation-ready. English is the only shipped locale today; adding a language needs no code changes. This
guide is the map + the cross-cutting workflows; per-area mechanics live in the colocated docs it points to (single
source — don't restate them).

## Architecture (where things live)

- **Runtime** — `apps/desktop/src/lib/intl/` (read its [`CLAUDE.md`](../../apps/desktop/src/lib/intl/CLAUDE.md) +
  `DETAILS.md`). One locale source (`getLocale()`) feeds two consumers:
  - the **message runtime** (`messages.svelte.ts`): `t(key, params?)` (catalog + ICU via `intl-messageformat`),
    `getMessage(key)` (raw lookup, no ICU), `setLocale()`, the locale-version rune, the compiled-message cache;
    `Trans.svelte` for sentences with an inline interactive component; `keys.gen.ts` (the generated `MessageKey` union,
    never hand-edit).
  - the **locale-aware formatter layer** (`locale.ts`, `number-format.ts`, plus dates in
    `$lib/settings/format-utils.ts`): numbers, file sizes, dates — all per OS region.
- **Catalog** — `apps/desktop/src/lib/intl/messages/en/<area>.json`, one file per feature area (read its
  [`CLAUDE.md`](../../apps/desktop/src/lib/intl/messages/CLAUDE.md) + `DETAILS.md`). ~2,050 keys across 28 files. Key
  prefix ↔ filename is 1:1 (`settings.fsWatch.title` → `settings.json`). `common.json` holds genuinely shared strings.
  ARB-style `@key` sibling entries carry the translator `description` (+ `placeholders` map, + optional `screenshot`),
  stripped before the runtime/codegen ever see them.
- **Enforcement** (checks, all green on every build):
  - `cmdr/no-raw-user-facing-string` (ESLint, `apps/desktop/eslint-plugins/`): flags a hardcoded string in a known sink
    (`title`/`label`/`placeholder`/`aria-label` props, `addToast(...)` args, `.svelte` text) in any migrated area. The
    enforced-area list is the whole frontend now; its exclusion ledger is empty.
  - `desktop-message-key-naming` + `desktop-message-keys-fresh` (Go, `scripts/check/checks/`): key shape
    `area.feature.leaf` against a closed area set, and `keys.gen.ts` freshness.
  - `desktop-message-keys-unused` (Go): error-level orphan check. A catalog key never referenced anywhere in
    `apps/desktop/src/` is dead translation work (real money once human translators are involved), so it fails the
    build. Runtime-built keys are carried by a closed, documented dynamic-prefix allowlist; canonical detail lives in
    `apps/desktop/src/lib/intl/messages/DETAILS.md` § Dead-key honesty + the orphan check.
  - `desktop-i18n-stale` (Go, nickname `i18n-stale`): WARN-only. Flags a non-`en` translation whose stored
    `@key.sourceHash` no longer matches the current English value's hash (the English source changed since the string
    was translated) — plus a missing hash, a removed English source, and a stale key still marked `reviewed: true`.
    English-only today, so it's a no-op until a real locale lands; it never blocks the build (escalating to a
    release-blocking error is a later policy choice). Detail: `messages/DETAILS.md` § `@key` schema (`sourceHash` /
    `reviewed`).
- **Maintenance tooling** — two shared, language-agnostic helpers the pseudolocale generator and the locale checks build
  on:
  - `apps/desktop/scripts/i18n-catalog-lib.js` (M0): load a locale's merged catalog (messages vs `@key` metadata split),
    enumerate locales/keys, parse a message to its ICU structure (`parseMessage` → placeholders, `<tag>`s, plural/select
    categories, `ok`/`error`) via the runtime's own `intl-messageformat` engine, and `sourceHash(value)` (the 7-char hex
    source hash). The single source for "what tokens/structure does this message have".
  - `apps/desktop/scripts/i18n-locale-check-lib.js` (M2): the reusable per-locale check scaffolding every locale check
    (stale + the future parity/ICU/plural/key checks) follows — `localesToCheck()` (non-`en` locales),
    `loadBaseCatalog()`, the `newFindings` accumulator, `reportFindings` (honest per-locale report), and
    `runLocaleCheck` (the load-en-once → inspect-each-locale → report loop), plus the
    `EXIT_CLEAN`/`EXIT_ISSUES`/`EXIT_ERROR` exit-code contract that the `desktop-i18n-*` Go wrappers map (exit 1 → WARN,
    other non-zero → ERROR). A new locale check is a thin script over `runLocaleCheck` + a Go wrapper modeled on
    `desktop-i18n-stale.go`. The file header documents the pattern in full.

## Add or change a user-facing string (the everyday task)

1. Add the message to the right `messages/en/<area>.json` with a semantic key `area.feature.leaf` (lowerCamel) and a
   sibling `@key` with a translator `description` (+ `placeholders` if it interpolates). Pick the area = the feature
   dir.
2. Resolve it at the call site: `t('area.feature.leaf', { … })` for static/plural/select copy; `<Trans>` when a
   component sits mid-sentence; `getMessage(...)` only for the error pipeline (see Gotchas). Never hardcode the literal.
3. Run `pnpm intl:keys` to regenerate `keys.gen.ts` (the typed `MessageKey` union; a missing key is a compile error, an
   unused catalog key is a dead-key warning).
4. Keep the area's `*-i18n-parity.test.ts` green (it pins base-en output byte-for-byte). A copy edit changes the catalog
   value AND the parity test together.

Conventions: ICU message **values** double every apostrophe (`''`); `@key` metadata (descriptions/placeholders) uses
**normal** apostrophes (it's not ICU). Embed counts as preformatted `*Text` string params (formatting is single-sourced
in `$lib/intl`), passing the raw integer alongside only to drive `plural` selection. Full `@key` description guidance +
the litmus test: `messages/DETAILS.md` § `@key` metadata schema.

## Add a new locale (future localization)

No code changes — the whole point of the readiness work. Create `messages/<locale>/<area>.json` files mirroring `en/`
(same keys, translated values), wire them into the catalog loader's glob, and `setLocale(locale)` switches at runtime
(reactive: markup re-renders; plain-`.ts` reads are snapshots). Dates/numbers/sizes already follow OS region via the
formatter layer.

### Locale-format convention (BCP-47, language-base fallback)

The locale VALUE is the full BCP-47 tag from `getLocale()` (`en-US`, `de-DE`, `pt-BR`); the formatter layer already uses
it for numbers, sizes, and dates. Don't change that.

Catalogs are keyed by BCP-47, with the **language base** as the universal fallback:

- `en` is the base (the source locale) and the FINAL fallback.
- A **language-base** catalog (`de`, `pt`) holds a full set for that language.
- A **region-variant** catalog (`en-GB`, `pt-BR`) holds only the overrides that differ from its base, or a full set.

**Resolution** for an OS locale `xx-YY`: try catalog `xx-YY` → try `xx` → fall back to `en` → the key itself (so a
missing key is visible, never a crash). The runtime's per-key `resolveRaw` already does base-language fallback today;
British English, for example, is `add an en-GB catalog` and the resolver prefers it over `en`.

The full per-OS-locale → catalog RESOLVER (the dir-selection layer that picks which `messages/<locale>/` to load, plus a
language selector) lands with the first real locale; today the loaded catalog is hardcoded to `en`. This convention
keeps the seam clean so that follow-on is a small, contained change. Note BCP-47 tag NAMES (`en`, `en-GB`, `pt-BR`) are
format identifiers, not translatable content.

## Leading translator agents (the localization workflow)

Translation is agent-driven + human-reviewed. Set an agent up for excellence with three inputs, kept separate:

- **Per-string context** = the `@key` `description` + `placeholders` (surface, trigger, constraints, do-not-translate
  tokens, plain-language placeholder meanings). Already authored for every key.
- **Per-language style guide** (you write, once per language) = tone, voice, formality, terminology/glossary. NOT
  per-string — never repeat tone on every key.
- **One ICU instruction** in the agent's system prompt: preserve the placeholder / `plural` / `select` syntax, translate
  only the human text, and use the TARGET language's CLDR plural categories (which can differ from English's two). ICU
  is a formatter, not a translator: it selects + fills; all grammar correctness comes from the authored per-locale
  strings.
- **(Optional) screenshots** (see below) give visual context that replaces describing layout in prose.

Principle 6 applies: translated locales meet human eyes, so the agent pipeline must include human review — "agents
translate, scripted pipeline" is not a license to ship unreviewed machine copy.

## Pseudolocale (`en-XA`)

The pseudolocale is a generated locale that takes each English value and produces a deterministic, accented,
~+40%-longer string while preserving every `{placeholder}`, `<tag>`, and ICU `plural`/`select`/`#` structure, plus a
`@key.sourceHash` per key. It exists for two jobs:

- **Overflow testing.** The accented, deliberately-longer text makes clipping and layout breakage obvious when the app
  is driven in `en-XA` and screenshotted. Driving the app in `en-XA` needs the runtime locale resolver (the
  dir-selection layer that lands with the first real locale; see § Add a new locale); the screenshots driver gains a
  `--locale` axis then. Today `en-XA` is generated for the checks and future overflow, not yet rendered in-app.
- **Check fixture.** It's a complete, structurally-faithful non-`en` locale, so the stale check and the
  parity/ICU/plural/key checks can run against a real locale before any human translation exists.

Regenerate with **`pnpm i18n:pseudo`** (root) or in the desktop app. It's deterministic: same English in →
byte-identical pseudo out, so re-running is a no-op diff.

The generator (`apps/desktop/scripts/gen-pseudolocale.js`) takes two transform paths, matching the runtime's `t()` vs
`getMessage()` split (see [`../../apps/desktop/src/lib/errors/CLAUDE.md`](../../apps/desktop/src/lib/errors/CLAUDE.md)):
walk the ICU AST and transform only literal text nodes for normal keys, and a letter-run tokenizer that leaves every
`{token}`, literal `<…>`, and markdown char untouched for the raw `errors.*` family. Both reuse the M0 helper
(`i18n-catalog-lib.js`); the test asserts `parseMessage(pseudo)` token sets equal the English source's for every key.

What's tracked vs not:

- The **full `messages/en-XA/`** dir is **gitignored + regenerable** (like the screenshot PNGs); it's derived from `en/`
  and churns on every copy edit.
- A **small, committed, hand-verifiable fixture** lives in `apps/desktop/test/fixtures/i18n-pseudolocale/` (an `en/` +
  `en-XA/` pair covering a plain label, a placeholder, a tag, a plural, a select, a multi-placeholder sentence, and a
  raw error). The stale + locale-validation check tests run against THIS fixture, not the gitignored full dir.
  Regenerate it with `node apps/desktop/test/fixtures/i18n-pseudolocale/gen-fixture.js` after changing the generator's
  transform.

## Screenshots

Each `@key` can carry a `screenshot` filename (in `messages/screenshots/`) showing the string in the running UI; one
screenshot serves many keys. The harness is shipped and re-runnable: `pnpm i18n:shots` captures fresh screenshots and
rewrites the `@key.screenshot` couplings in one command. Reach for it after a UI change.

How it works (mechanism lives with the code — see
[`apps/desktop/src/lib/intl/messages/DETAILS.md`](../../apps/desktop/src/lib/intl/messages/DETAILS.md) § Screenshots,
plus the headers of `apps/desktop/scripts/i18n-capture.js` and `couple-screenshots.js`): a capture-mode instruments
`t()`/`getMessage()`/`<Trans>` to record which keys each surface resolves, behind the build-time `__CMDR_I18N_CAPTURE__`
define so it's dead-code-eliminated from prod (verified prod no-op). A Playwright driver targets each window via the
`tauri-playwright` fork's label-aware native screenshot and writes `screenshots/capture-report.json` (surface → keys).
The coupler (`couple-screenshots.js`) reads that report and writes `@key.screenshot` line-surgically, plus a tracked
`screenshots/coverage-report.md` showing per-area what's coupled and which keys aren't.

Two kinds of coupling raise coverage honestly:

- **Direct**: a key that rendered on a captured surface gets that surface's screenshot — a precise capture of its own
  UI.
- **Representative**: for a key still uncoupled after the direct pass that matches a curated prefix
  (`REPRESENTATIVE_SCREENSHOTS` in `couple-screenshots.js`), the coupler writes a STAND-IN screenshot of the same
  panel/toast/dialog where the string appears, plus a `@key.screenshotNote` explaining the mapping (e.g. "this shows a
  different error, but your string is the title/explanation/suggestion in this same pane"). This sets translators up for
  success: it covers families with no captured surface of their own (the dynamic `errors.*` tail, AI/cloud states, SMB
  reconnect, the crash-report dialog, the shortcuts window) AND shrinks the number of distinct images a translator
  loads. Direct always wins (a stand-in never overwrites a precise capture), and a representative only points at an
  image the run actually produced. The coverage report counts direct vs representative separately so a stand-in is never
  mistaken for a precise capture.

PNGs are gitignored and regenerable; what's TRACKED is `capture-report.json`, the coverage report, and the
`@key.screenshot` / `@key.screenshotNote` refs in the catalogs. A warn-only check (`message-screenshots-fresh`) flags
when the couplings drift from the report — never a build failure, since screenshots are an optional translator aid
(description + key are enough to ship a locale). Coverage is partial until the driver covers the full surface inventory.

## Gotchas (hard-won)

- **The error pipeline bypasses ICU.** Listing/git/provider/write-error prose resolves via `getMessage()` (raw lookup),
  NOT `t()`: its `{system_settings}` `expandSystemStrings` tokens and `esc()` HTML entities collide with ICU's
  brace/apostrophe grammar. So error catalog values use NORMAL apostrophes and keep their composition/escaping logic in
  `$lib/errors/` (`compose.ts`). Don't route them through `format()`. The frozen golden fixture
  (`$lib/errors/__fixtures__/friendly_error_golden.json`) is the parity net — never regenerate it.
- **`<Trans>` tag names must differ from param names** — a tag named like a param silently overwrites the param value.
- **A literal `<` in a catalog value is parsed as a markup tag** — ICU throws `UNCLOSED_TAG` at render. Escape it
  (`'<'`); this bit the `<dir>`/`<no perms>` size placeholders.
- **`keys.gen.ts` is generated** (`pnpm intl:keys`); never hand-edit. Dynamically-built keys (the error factories'
  ``getMessage(`errors.listing.${reason}.title`)``) read as "dead" to the static scanner; the error-level
  `desktop-message-keys-unused` check carries them via a closed dynamic-prefix allowlist so they don't fail the build.
  Both layers are documented in `messages/DETAILS.md` § Dead-key honesty + the orphan check.
- **Formatter parity**: en-US output is byte-identical to the pre-i18n code EXCEPT raw-byte triads (now comma-grouped,
  was a U+2009 thin space); human-friendly sizes use `useGrouping: false` to stay identical. Net `en-us-parity.test.ts`.
