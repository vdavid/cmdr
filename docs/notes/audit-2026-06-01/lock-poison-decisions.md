# Lock-poison triage decisions

Triage of the 47 production sites flagged by `./scripts/check.sh --check lock-poison` on branch `audit-2026-06-01`.
Rubric: the module doc of `apps/desktop/src-tauri/src/ignore_poison.rs` (recover-by-default for simple value stores;
abort only when the lock guards a real cross-field invariant or a state machine passing through a temporarily-illegal
intermediate state).

Result: **46 recover, 0 abort, 0 flagged-uncertain, 1 opt-out** (the lone opt-out, `test_fixtures.rs:251`, is not a std
lock at all — it's a `gix` tree-writer whose `.write()` matched the checker's regex). Every flagged std `Mutex`/`RwLock`
guards an independent value store (a `Vec`, `HashSet`, `HashMap`, `Option<T>` slot, an `Instant` debounce cache, or a
once-consumed channel `Sender`), so a panic mid-operation can't tear a cross-field invariant. The one genuine
state-machine lock in this area — `indexing::state::INDEXING` (`Mutex<IndexPhase>`) — is **not** in the worklist; it
already aborts deliberately via `.expect("INDEXING lock poisoned")` / `.map_err(...)`. Reconciling `state.rs` therefore
means making its other flagged locks (`READ_POOL`, `PENDING_SIZES` — plain `Option<Arc<…>>` slots) recover, so the file
is internally consistent: state machine aborts, value stores recover.
